# 启动阶段的飞轮效应

## 背景

连续好几次打开项目，在启动阶段就失败了——看到一堆计划文件，不知道从哪开始，注意力就飘走了。这是典型的启动摩擦力过大。

## 飞轮模型

```
静止 → 第一圈（最费力） → 加速 → 自转（靠惯性）
```

应用到学习：

| 阶段 | 飞轮状态 | 策略 |
|------|----------|------|
| 现在 | 静止 | 让别人帮你做启动工作（创建文件、编译通过） |
| 第一圈 | 开始转动 | 改一行代码，看效果变化 |
| 加速 | 惯性起来 | 自己写 build 方法，每写一个就能切换看到 |
| 自转 | 停不下来 | 主动去搞 Chapter 2 |

## 我的理解

拖延的本质不是懒，是**启动能量太高**。解决方案是降低启动门槛：

- **体力活外包**：文件创建、CMake 配置让工具/别人做
- **从"改"开始**：在已有代码上改，比从零写心理负担小得多
- **快速反馈**：每改一点就能看到效果，形成正循环

## 关键洞察

> 学习的核心是你写代码，但启动阶段的文件创建纯粹是体力活。

区分"学习性工作"和"体力活"：
- 学习性工作（必须自己做）：理解概念、写 build 方法、调试
- 体力活（可以外包）：创建文件骨架、配置 CMake、写 boilerplate
